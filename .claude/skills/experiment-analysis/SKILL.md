## ▶ 前置理解步骤（禁止"盲读数字"）

**核心原则**：在不理解"这组实验在测什么"、"某指标怎么算出来的"、"某参数意味着什么"之前，
**禁止**对数据给出任何判断。必须先读代码搞清楚，再开始分析。

### 触发理解步骤的信号

当出现以下任一情况，Agent **必须**先读代码再继续：

1. **不知道实验组在控制什么变量**
   - 例：看到 `gamma_study_slenderer_low` 这种命名无法确定 γ 扫描范围
   - 例：看到 `rawdenoise_study/alpha` 与 `rawdenoise_study/binning` 不知道两者区别
   - → 读 `automation_config.yaml` 或 `ISPpipline/ISP_main.py` 看参数定义
   - → 读 `ISPpipline/Pipeline/` 下对应模块看具体实现

2. **不知道某指标是怎么算出来的**
   - 例：看到 `Compare_Delta_SNR_dB` 为负，不确定正/负方向代表什么
   - 例：看到 `Quadrant=denoise_shift` 不清楚四象限的阈值定义
   - → 读 `probe_analysis/probe_ana_cmp.py` 相关函数
   - → 读 `probe_analysis/analysis_method.md` 与 `probe_analysis/probe_method_now.md`

3. **不知道探针采集在哪个 ISP 阶段**
   - 例：看到 `CCM-Gamma_timeseries.csv`，不确定是 CCM 之后 / Gamma 之前还是反过来
   - → 读 `ISPpipline/pipeline_probe.py` 看探针插入点
   - → 读 `probe_meta.json` 确认 `domain` / `stage_name`

4. **不知道 pyVHR 某参数含义**
   - 例：`RGB_LOW_HIGH_TH=(10, 230)` 与 `(75, 230)` 为什么不同
   - → 读 `pyVHR_run_on_video/analyze_with_pyvhr.py` 与 pyVHR 源码
   - → 读 rppg-analysis 规则的标准调用方式

5. **指标含义与直觉不符**
   - 例：某组 `BPM_MAE` 接近 0 但 `SNR_dB` 为负（看起来矛盾）
   - → 必须回到计算源码，不能猜测"大概是 X 原因"

### 执行规范

- 读代码时**用工具，不用猜**：Read 对应实现文件、Grep 相关变量名、查 `probe_meta.json`
- 读完必须**显式声明**理解结果，例如：
  > 已确认 `Compare_Delta_SNR_dB = 探针2_SNR − 探针1_SNR`（见 probe_ana_cmp.py:306），
  > 因此负值表示经过该 ISP 模块后 SNR 降低。
- 若读完代码仍有不确定（例如实现本身有 bug 或文档与代码不符），**必须向用户确认**，不能自行补完语义
- **禁止**基于"变量名字面意思"或"常识"下结论——命名可能过时、可能写错、可能与实现不一致

### 反例（必须避免的行为）

- ❌ 看到指标 `Peak_Prominence_Ratio = 1.2` 就说"峰不够突出"——不知道阈值规则
- ❌ 看到目录名 `rawdenoise_study/alpha` 直接假设 α 是滤波强度——必须读实现
- ❌ 看到某受试者 SNR 异常低就说"该受试者数据质量差"——可能是 ROI 皮肤阈值与其他组不一致
- ❌ 引用"论文中常用的 X 方法"时未确认本项目代码是否是标准实现

### 节约时间的短路规则

- 同一份代码在同一次会话中已读过 → 不重读，但必须在使用时引用行号
- 简单指标（Mean / Std / PCC）的标准定义 → 可不查源码
- 涉及**阈值 / 频带 / 归因逻辑 / 自定义指标**的 → 必须查源码

## ▶ 报告输出位置与格式（强制）

### 输出位置

**每一次分析的最终报告，必须直接写到对应实验结果目录下**，而不是仅以聊天消息形式存在。
这保证报告与数据同地存放、可长期溯源。

| 分析对象 | 报告写入目录 | 文件名建议 |
|---------|-------------|----------|
| `rPPGanalyze_res_plots/sensitivity_analysis/<ISP模块>_study/` | 同目录 | `ANALYSIS_REPORT.html` (主) + `ANALYSIS_REPORT.md` (源) |
| `probe_analysis/probe_analysis_result/<实验类型>/<实验组>/` | 同目录 | `ANALYSIS_REPORT.html` (主) + `ANALYSIS_REPORT.md` (源) |
| 跨实验对比 | 最主要实验目录 | `ANALYSIS_REPORT_<对比名>.{html,md}` |

**输出格式规定**：

- **主报告必须是 HTML**：HTML 打开即是渲染后的样式（字体层级、表格、代码块一目了然），避免读者面对 Markdown 源码（`#`、`|`、`**`）。
- **同目录同步保留 .md 版本**：用于 diff / git / grep 检索。两个文件内容等价，仅格式不同。
- HTML 样式要求：
  - 最小行高 1.6、表格斑马纹、代码等宽字体
  - 标题 h1/h2/h3 字号依次 28/22/18 px 左右
  - 全文最大宽度 960 px，居中
  - 可以用内联 `<style>` 块（单文件自包含，便于复制分享），不依赖外部 CSS/JS
- 同一目录若已存在报告：**不要直接覆盖**，用带日期的命名 `ANALYSIS_REPORT_YYYYMMDD.{html,md}`，保留历史版本
- 支撑报告的**原始配对统计表 / 中间计算脚本**同目录另存（例：`paired_stats.csv`、`aggregate_stats.py`），使报告可复现
- 报告中引用"上节工具输出"、"刚才计算的结果"是**禁止行为**——必须引用同目录下的持久化产物

### 报告格式（精简优先）

交付的分析报告应面向"研究者/同事快速理解"，**不是 Agent 的工作日记**。强制遵守：

1. **单页思维**：正文 ≤ 300 行 Markdown。需要长表格时单独存 CSV 并在正文引用。
2. **顶部放核心结论**：一句话核心结论 + 最多 3 条 bullet 的关键发现，放在文档最前。
3. **关键指标必须给"含义一句话"**：首次出现的每个关键指标（SNR_dB / Compare_Delta_SNR_dB / MAE / PCC / 四象限名 / 自定义字段等）**都要给一句通俗解释**，让不熟悉本项目的读者也能看懂。例：
   > `SNR_dB`：心率主峰（0.65–4.0 Hz）能量与背景噪声能量之比，越大信号越纯净。
4. **绝对值与 Δ 值必须并列呈现**：只给 Δ 会让人不知道当前水位（"降了 0.7 dB 是从 9 dB 到 8.3 dB 还是从 2 dB 到 1.3 dB？影响级别完全不同"）。规则：
   - 每个关键指标表格必须**同时**给出**各组绝对值（mean 或 median）** 和 **Δ vs baseline**
   - baseline 组的绝对值必须显式列出作为参照基准
   - 推荐列顺序：`group | 绝对值 mean±std | Δ median | Δ pos/neg | |t|`
   - 支撑 CSV 也必须同时包含两类列（`abs_mean` / `abs_std` / `delta_*`）
5. **多指标平等展示，禁止替读者筛选**：如果报告中一张表已经展示了某指标 A（如 SNR）的 `Δ / pos/neg / |t| / 判定` 完整四件套，则**同表中其他核心指标**（至少 MAE 或 RMSE 之一）也**必须给出完整四件套**，不得仅留绝对值而省略 Δ/判定。理由：
   - 读者需要自己判断"SNR 受损"和"MAE 是否传导"之间的关系（denoise_shift 现象就藏在这里）
   - Agent 不是裁判，不能预先判断"这个指标看着没变化就不展示"——这违反"证据完整性"原则
   - 例外：若某指标在所有实验组上 Δ 均为 0 且 |t|<0.5，可在同一表格下方用 caption 一句说明"MAE 全组未变化"而不展示列，但必须**显式声明**
6. **表格优先于散文**：跨受试者 / 跨参数的对比必须用表格呈现，每列标注单位，关键列用**加粗**。
7. **结论必须分级**：每个结论句后标注置信度（强 / 中 / 弱 / 不可结论），按第 5.2 节判定。
8. **无冗余章节**：机理解释可省略已经在第 3 节讲过的套话；报告只写"本实验中的具体观察"。
9. **结尾给下一步**：最后 3–5 行列出"待补实验"或"下一步验证"，避免停在空结论。

## 0. 第一性原理：永远先回答这两个问题

任何一次分析、一份报告、一张图、一段结论，开篇都必须显式锚定到下面两个研究目的之一或两个：

1. **机理目的**：该 ISP 模块（或参数）是**破坏**还是**增强**了 rPPG 生理信号？**为什么**？
   - 「破坏/增强」必须落到可量化的指标变化（SNR_dB、BPM 误差、HR_Band_Ratio、PCC 等）
   - 「为什么」必须给出物理/数学解释，对应 ISP 模块的算法假设与 rPPG 算法假设的耦合关系
2. **创新目的**：基于观察到的机理，能否提出一个 **ISP 算法层面的可改造点**，让 rPPG 信号增强 / 抗干扰 / 抗欺骗？
   - 创新点必须 **落到 ISP 模块**（不是 rPPG 算法侧），且与本项目可实现路径兼容（白盒 Python ISP）

**禁止行为**：
- 禁止只罗列指标数值而不解释机理（"SNR 降低了 2 dB" → 必须说为什么）
- 禁止用 rPPG 算法的术语解释 ISP 现象，反之亦然（要明确"是 ISP 改变了输入分布"还是"是 rPPG 算法对该分布敏感"）
- 禁止在没有跨受试者验证（N ≥ 5）的情况下下"该模块普遍有害/有益"的结论

## 1. 分析对象与数据源对齐

本项目有**两条平行的实验线**，分析时必须明确当前对象属于哪一条，避免把指标混着读：

| 实验线 | 产物根目录 | 视角 | 主指标 |
|--------|-----------|------|--------|
| **探针白盒分析** | `probe_analysis/probe_analysis_result/<实验类型>/<实验组>/<subject>/` | ISP 流水线**内部**逐节点信号质量 | SNR_dB、AC_DC_Ratio、HR_Band_Ratio、Compare_Delta_SNR_dB、四象限 |
| **端到端 rPPG 黑盒分析** | `rPPGanalyze_res_plots/<实验类型>/<ISP参数>/<rPPG算法>/group_*/` | ISP **输出视频**送入 pyVHR 后的最终 BPM 误差 | RMSE、MAE、MAX、PCC、CCC、SNR（pyVHR 的） |

**对齐原则**：
- 探针 SNR_dB 与 pyVHR 输出的 SNR **量纲与频带不同**，禁止直接数值比较；但**趋势必须一致**，若不一致则需要单独解释（通常是 pyVHR 内部的滤波/ROI 差异引入的）。
- 探针的 BPM_Estimate（Welch 峰值）与 pyVHR 的 bpmES（窗口估计 + 后滤波）允许有偏差，但**变化方向**应一致。
- 同一受试者、同一实验组，必须先看探针流水线里**哪一节点**开始变差，再去看端到端误差是否在该节点引入时塌方——这是机理归因的核心证据链。

## 2. 分析的分层视角（三层，不可跳级）

每份分析报告必须按以下顺序逐层展开，不允许从最高层直接下结论：

### Layer 1 · 单受试者 × 单实验组 × 单模块

- 最小粒度：一次 `run_analysis()` 的输出目录（`metrics_summary.csv` + 4 张 PNG + `raw_channels/`）
- 必看产物：
  - `metrics_summary.csv` 的 **12 个分区**全部读（基础信息 / 核心判决 / 时域 / 频域 / HRV / 通道色彩 / 锚 A / Layer1 自诊断 / Layer1 锚 A / Layer1 锚 B / 四象限 / 跨探针）
  - `timeseries_comparison.png`：看 AC 波形是否还能识别出心跳节律
  - `psd_comparison.png`：看心率主峰（0.65–4.0 Hz 内）是否依然突出
  - `spectrogram.png`：看心率频带能量是否**整段稳定**（断续=皮肤检测丢帧或 AC 被压制）
  - `raw_channels/g1g2_consistency_*.png`（RAW 域必看）：G1/G2 一致性偏离→sensor 噪声或 demosaic 串扰
- 输出要求：必须能用一句话回答"在这名受试者上，该模块通过什么物理路径改变了信号"

### Layer 2 · 跨受试者聚合（同一实验组）

- 至少 N ≥ 5（建议 N ≥ 10，本项目当前 11 名受试者）
- 必算量：
  - 各核心指标（SNR_dB / BPM_MAE / Compare_Delta_SNR_dB）的 **median + IQR** 而不是 mean ± std（个体差异通常非高斯）
  - 四象限的**人数分布**（例如 11 人中 8 人落入 over_idealize、2 人 pure_harm、1 人 unclear）
  - 个体一致性：方向相同的占比（"该模块在 N/M 名受试者上 ΔSNR 同号"）
- 报告要求：
  - 若四象限分布跨过两个以上象限，必须**追问个体差异原因**（肤色、光照、运动量、心率范围、ROI 像素数）
  - 个体差异不能简单归为"噪声"，要尝试用 `metrics_summary.csv` 里的元数据（ROI_Skin_Pixel_Count、G_R_Ratio 等）做分层

### Layer 3 · 跨实验组（参数扫描 / 算法对照）

- 例如 Gamma ∈ {0.2, 0.4, 1.0, 2.2, 3.0, 4.0, 6.0}，或 Demosaic ∈ {bilinear, malvar, AHD, CV}
- 必看产物：
  - `all_groups_summary.json`：每组的 average_metrics + std
  - `all_groups_metrics_boxplot.png`：跨组分布对比（看中位线趋势 + 异常点）
  - `all_groups_barchart_of_average_errors.png`：均值条形图（结合 std 判定显著性）
- 分析要求：
  - 必须画出/口述参数 → 指标的**单调性**（单调、凸/凹、阈值跳变、双峰）
  - 找到**最优值**和**塌方点**（最优值附近的稳健带宽是创新点候选）
  - 对**单调或单峰**的结果，必须给出物理解释（例如 Gamma > 2.2 后 AC 被压缩 → SNR 下降）
  - 对**非单调或多峰**的结果，要警惕实验混杂因素（光照漂移、不同受试者占比、ROI 数量差异）

## 4. 关键指标的正确读法（禁止误读）

### 4.1 SNR_dB 的"绝对值 vs 相对变化"

- **不要看绝对值排名**：不同受试者基线 SNR 不同（4 dB 到 15 dB 都正常）
- **要看相对变化**：`Compare_Delta_SNR_dB`（同一受试者，模块前后），或同一受试者跨参数扫描
- **跨域比较**（RAW→RGB→YUV）必须用 z-score 后的对比指标（系统已做）

### 4.2 BPM_Estimate 的可信度门槛

- 单窗口 BPM 估计不可信，**至少要求**：HR_Band_Ratio > 0.15、Peak_Prominence_Ratio > 2.0、SNR_dB > 0
- 否则 BPM 是"噪声峰"，不能用来下"模块改变心率估计"的结论
- 看 `Valid_SNR_Window_Count`：有效窗口 < 总窗口 50% → 该探针该模块结果**整体不可信**

### 4.3 四象限分类的解读规则

| 象限 | 解读 | 论文叙述模板 |
|------|------|-------------|
| ideal_enhance | **创新的反向案例**：保留并采用此处理 | "模块 X 在受试者上一致落入此象限，建议保留" |
| over_idealize | **典型 ISP 陷阱**：视觉好看但 rPPG 伪信号 | "X 在 Y 名受试者上美化但偏离真值，需替换为 rPPG 友好版本" |
| denoise_shift | **机理研究金矿**：起点变了但更靠近真值 → 揭示某些 ISP 假设其实有利 | "X 移动了 RAW 基线，但靠近 PPG GT，提示 RAW proxy 本身存在系统偏差" |
| pure_harm | **删除候选**：双向变差 | "X 在所有受试者均损害信号，论文消融时强烈建议关闭" |
| neutral / unclear | 慎下结论 | 检查数据量与显著性 |

- **跨受试者多数票**：若 N=11 中 ≥ 7 人落入同一象限，该结论可写入论文；< 7 人则需要分层（按肤色/光照/心率）

### 4.4 HRV 指标（SDNN/RMSSD/pNN50/LF_HF）的边界

- HRV 计算依赖**峰检测正确性**——若 SNR < 6 dB 几乎一定错，HRV 数值**不可单独使用**
- 本项目中 HRV **仅作为辅助一致性证据**：HRV 在模块前后剧烈变化 → 印证主信号被破坏；但**反过来不成立**（HRV 没变不等于信号没受影响）

### 4.5 RAW G1/G2 一致性

- 物理上 G1/G2 是同色像素，理想 `RAW_G1_G2_PCC` → 1
- 显著偏离 1 的原因优先级：① demosaic 串扰 ② sensor 通道非均匀 ③ ROI 偏移导致 Bayer 错位
- 当 G1/G2 PCC 突然从 0.99 掉到 0.7：通常是 demosaic 模块的副作用，**这是 demosaic 影响 rPPG 的关键证据**

## 5. 跨受试者统计与可信度

### 5.1 统计方法

- 配对设计（同一受试者、同一录像，仅 ISP 参数不同）→ 用 **paired test**：
  - 正态：paired t-test
  - 非正态：Wilcoxon signed-rank
- 多组比较（≥ 3 参数）→ **Friedman test + post-hoc Wilcoxon with Bonferroni**
- 报告必须包含：N、median + IQR、p-value、effect size（Cohen's d 或 r）

### 5.2 置信度分级（必须在结论中标注）

| 级别 | 条件 | 写作措辞 |
|------|------|---------|
| **强结论** | N ≥ 10，方向一致 ≥ 80%，p < 0.01，效应量中等以上 | "X 显著降低 SNR" |
| **中等结论** | N ≥ 5，方向一致 ≥ 60%，p < 0.05 | "X 在多数受试者上倾向于降低 SNR" |
| **弱结论 / 假设** | N < 5 或方向不一致 | "初步观察 X 可能降低 SNR，需扩大样本验证" |
| **不可下结论** | 数据缺失、ROI 不一致、四象限 unclear ≥ 50% | "数据不足以判定 X 的影响" |

### 5.2.1 三元组判定矩阵（每个数据行必须套用）

**核心原则**：**单看任一个量都会误导**。对每一个指标列（SNR / MAE / RMSE / PCC ...），
必须同时读 `Δ 绝对值` + `pos/neg 方向一致性` + `|t|`，按下表判定并**直接写入报告的每个数据行**。

| Δ 绝对值 | pos/neg 一致性 | \|t\| | 判定 | 行动 / 措辞 |
|---|---|---|---|---|
| 大 | 高（≥80%） | 大（≥3.17） | **强证据 · 模块造成** | "X 显著改变该指标，强结论" |
| 大 | 高（≥80%） | 中（2.23~3.17）| **中等证据 · 模块造成** | "X 在多数受试者上改变该指标，中等结论" |
| 大 | 高（≥60%） | 小（<2.23） | **真实效应存在但被个体差异稀释** | "倾向性明确但未达显著，**扩样本**或**分层分析**" |
| 大 | 低（对半） | 小 | **双峰 / 方向抵消** | "子群体反向，必须**分层**（按肤色/心率/光照）" |
| 小 | 任意 | 小 | **模块无效应（接受 H₀）** | "X 对该指标无影响，本身是有效结论" |
| 小 | 任意 | 大 | **大样本病**（微小效应被放大） | "统计显著但无实际意义" |

**阈值约定（N=11, df=10）**：
- Δ "大" / "小" 的界线要**结合指标量级**判定（SNR 以 0.3 dB 为参考线，MAE 以 0.02 bpm 为参考线），单位不同不可跨指标比较
- pos/neg "高一致性" = ≥ 9/2（80%）；"中一致性" = ≥ 7/4（60%）；"对半" = 5/6 或 6/5
- \|t\| 阈值：2.23 → p<0.05，3.17 → p<0.01（df=10）

**关键澄清**（必写进报告以避免读者误读）：
> **\|t\| 小不代表 SNR / MAE 数值本身不可信**，只代表"观察到的改变方向不能归因于模块"。
> "无效应"与"数据错"是两件事——"模块对该指标无影响"本身就是一个**可写进论文的有效结论**。

### 5.2.2 报告中必须标注"判定"列

每个结果表格的最后一列必须是 **判定**（`强证据` / `中等` / `倾向但未显著` / `分层` / `无效应` / `大样本病`），
按 5.2.1 矩阵给出。读者一眼就能知道该行该怎么对待，而不是自己算 Δ × 方向 × |t|。

### 5.3 混杂因素清单（每份报告必须自查）

- ROI 像素数差异（不同 `rgb_low_th`、不同肤色导致皮肤 mask 大小不一致）
- 录制时长不同（窗口数差异）
- 心率范围差异（静息 vs 运动后）
- 受试者光照条件差异（baseenv 是否真的"基线"了）
- pyVHR 参数差异（必须确保 winsize / method / RGB_LOW_HIGH_TH 跨组一致）

## 6. 创新点发现方法论

本研究的"创新"具有明确定义：**在 ISP 算法层面引入一个修改，使 rPPG 信号在统计意义上改善**。
寻找创新点时必须按以下顺序：

### Step 1 · 从"反常现象"出发

- 找 `metrics_summary.csv` 中**违反直觉的指标**：
  - 视觉好看的参数（高对比、强降噪）反而 SNR 下降 → over_idealize 象限
  - "非标准"参数（identity CCM、γ=1.0、bilinear demosaic）反而 SNR 上升 → 创新候选
- 找**单峰最优**而非"越多越好"的参数：最优值附近就是 ISP 算法的甜点带

### Step 2 · 确认机理而非巧合

- 提出可证伪的假设："是因为 X 导致 Y"
- 在另一个独立实验组上验证（不同受试者、不同光照）
- 用探针流水线**逐节点定位**：从 X 的输入和输出处取信号，看 Y 是否能复现

### Step 3 · 提出 ISP 算法改造

创新方向（按本项目可行性排序）：

1. **ROI 自适应**：在皮肤 ROI 内使用一套参数，非皮肤区使用另一套（最易落地）
2. **频带选择性**：保护 0.65–4.0 Hz 心率频带不被增强/抑制（中等难度）
3. **时序参数稳定化**：动态 ISP 参数（AWB / 自动 γ）的时间常数远大于心率周期（中等难度）
4. **目标函数重设计**：以 rPPG SNR 为目标反向优化 ISP 参数（高难度，但论文价值最高）
5. **跨模块协同**：如 Demosaic + Denoise 联合优化（高难度）

### Step 4 · 创新点的反向验证

- 必须设计**消融**证明该改造在去掉时性能退化
- 必须测试 **rPPG 算法泛化**（CHROM / POS / GREEN / LGI 上是否都有效，至少 3 个）
- 必须给出**失败模式**（什么受试者/场景下该改造反而有害）

### Step 5 · 防御性反思（避免假阳性创新）

- 改造收益是否在**误差棒重叠**内？→ 不是真创新
- 是否仅在单一受试者上成立？→ 数据不足
- 改造代价（计算量、视觉质量损失）是否大于 rPPG 收益？→ 需要 trade-off 表

## 9. 当未达到结论门槛时的强制行为

如果当前数据满足以下任一条件，**禁止**给出"该模块对 rPPG 有/无影响"的最终结论：

- N < 5
- 同一现象在 ≥ 30% 受试者上反向
- 四象限 unclear 比例 ≥ 50%
- 关键指标（SNR_dB / BPM_MAE）的 IQR > 中位数差异的 2 倍
- 跨实验组的 `pyVHR` 配置不一致（特别是 `RGB_LOW_HIGH_TH`）

正确做法是输出一份"**待补实验清单**"：列出还需要哪些受试者、参数、配置才能得到强结论。
